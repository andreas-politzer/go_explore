1. Exploratory Data Audit (Foundational Matrix)

Following a deep architectural profiling of the legacy spreadsheets inherited from the predecessor, the BI engine maps operations onto explicit entity relations. This strict metadata layout eliminates calculation anomalies during scaling:

Audited Structural Parameters:

  • 4 Relational Entities: retailers (B2B network attributes), products (product ledger dimension master), daily_sales (unaggregated core transaction journal), and methods (distribution channel identifiers).
  • Transactional Scale: A total database volume of 149,257 unique historical row records spanning historical execution files from January 2015 through July 2018.
  • Geographic Distribution: 21 active baseline international countries currently logging continuous distribution records (e.g., United States, Germany, France, Sweden, Japan, Australia, Canada).
  • Distributor Classifications: 8 distinct retailer categories mapped directly into Specialists (Golf Shop, Eyewear Store, Equipment Rental Store) and Generalists (Outdoors Shop, Sports Store, Department Store, Warehouse Store, Direct Marketing).
  • Channel Vectors: 7 operational ordering streams logged inside backend indices (Fax, Telephone, Mail, E-mail, Web, Sales visit, Special) to accurately gauge automated vs. manual operational patterns.
  • Calculated Metrics: The matrix does not provide a native total financial revenue field. Total sales value must be computed programmatically on row execution level utilizing vector mathematics: Quantity * Unit sale price.

3. Production KPI Matrix Blueprint

These 5 target metrics directly resolve the executive inquiries concerning European market entry scaling and B2B partner performance evaluation.

KPI 1: Total Revenue by Country (Baseline Portfolio Distribution)

Geo-Expansion
Strategic Business Value:

Isolates and ranks financial performance across the 21 active international territories. This establishes the absolute market volume baseline necessary to evaluate high-performing regions and construct reliable benchmarks for prospective expansion countries.

Analytical Formula:
Total Financial Revenue = SUM(Quantity × Unit sale price) GROUP BY Country
BigQuery SQL Engine Implementation:
SELECT r.Country, SUM(s.Quantity * s.Unit_sale_price) AS Total_Revenue FROM `go_explore_prod.daily_sales` AS s INNER JOIN `go_explore_prod.retailers` AS r ON s.Retailer_code = r.Retailer_code GROUP BY r.Country ORDER BY Total_Revenue DESC;

KPI 2: Revenue per Capita (Demographic Proportional Scale)

Geo-Expansion
Strategic Business Value:

Normalizes raw country performance against population vectors to remove demographic scale bias. By deriving a "Revenue per 1 Million Inhabitants" metric from mature markets (e.g., Sweden), we construct a scaling multiplier. This multiplier is cross-referenced with target market benchmarks to forecast expected market capture limits.

Ingested Target Macro-Economic Reference Data:
Target Expansion Country Population Metric (Eurostat) Nominal GDP (World Bank Baseline)
Norway5.5 Million$580 Billion USD
Czech Republic10.8 Million$330 Billion USD
Poland36.8 Million$860 Billion USD
Portugal10.4 Million$270 Billion USD
Analytical Formula:
Revenue per Capita = Total Country Revenue / Country Population (in Millions)
BigQuery SQL Engine Implementation:
SELECT r.Country, SUM(s.Quantity * s.Unit_sale_price) / ext.Population_Millions AS Revenue_per_Capita, ext.GDP_Billions_USD FROM `go_explore_prod.daily_sales` AS s INNER JOIN `go_explore_prod.retailers` AS r ON s.Retailer_code = r.Retailer_code LEFT JOIN `go_explore_prod.external_market_data` AS ext ON r.Country = ext.Country GROUP BY r.Country, ext.Population_Millions, ext.GDP_Billions_USD ORDER BY Revenue_per_Capita DESC;

KPI 3: Average Transaction Value (ATV by Retailer Segment)

Retailer-Analysis
Strategic Business Value:

Directly evaluates the purchasing capacity of specialized accounts versus broad general networks, resolving the primary query raised by the Head of Retailer Connections. This reveals whether high-intent specialty shops yield higher capital intake per transaction block than general stores.

Analytical Formula:
ATV = Total Segment Financial Revenue / Total Unique Transaction Blocks (Date || Retailer)
BigQuery SQL Engine Implementation:
SELECT r.Type AS Retailer_Segment_Type, SUM(s.Quantity * s.Unit_sale_price) / COUNT(DISTINCT CONCAT(CAST(s.Date AS STRING), '-', CAST(s.Retailer_code AS STRING))) AS Average_Transaction_Value FROM `go_explore_prod.daily_sales` AS s INNER JOIN `go_explore_prod.retailers` AS r ON s.Retailer_code = r.Retailer_code GROUP BY r.Type ORDER BY Average_Transaction_Value DESC;
⚠️ Data Architecture Constraint Warning (For Instructor Defending)

The string concatenation CONCAT(Date, '-', Retailer_code) serves as a critical macro-transaction proxy since a distinct, native transaction Order_ID is absent from the unaggregated journal ledger. If a partner locations commits multiple unique purchase orders within a single calendar date, the SQL syntax consolidates them into a singular daily financial unit. This architectural boundary is explicitly surfaced as part of the pipeline limitations during data defense.

KPI 4: Average Unit Sale Price (Segment Pricing Position Master)

Retailer-Analysis
Strategic Business Value:

Introduces the secondary validation vector required for a complete partner network assessment. While KPI 3 determines the total volume of daily orders, KPI 4 profiles the product tiers themselves. It verifies whether specialty distributors (e.g., Golf Shops, Eyewear Stores) successfully shift high-margin, premium equipment price ranges compared to general consumer networks.

Analytical Formula:
Average Unit Sale Price = AVG(Unit sale price) GROUP BY Retailer Type
BigQuery SQL Engine Implementation:
SELECT r.Type AS Retailer_Segment_Type, AVG(s.Unit_sale_price) AS Avg_Unit_Sale_Price, MAX(s.Unit_sale_price) AS Max_Unit_Sale_Price FROM `go_explore_prod.daily_sales` AS s INNER JOIN `go_explore_prod.retailers` AS r ON s.Retailer_code = r.Retailer_code GROUP BY r.Type ORDER BY Avg_Unit_Sale_Price DESC;

KPI 5: Web Channel Penetration Rate (Digital Optimization Speed)

Operations-Efficiency
Strategic Business Value:

Provides CEO Sabah with operational efficiency markers mapping organizational scalability. By isolating transaction rows generated via channel stream index code 5 (Web) against high-overhead manual methods (Fax, Phone, Mail), the dashboard measures the company's automated scaling velocity.

Analytical Formula:
Web Channel Penetration Rate (\%) = (Web-Generated Revenue / Total Revenue Across All Channels) × 100
BigQuery SQL Engine Implementation:
SELECT SUM(CASE WHEN s.Order_method_code = 5 THEN s.Quantity * s.Unit_sale_price ELSE 0 END) AS Web_Revenue, SUM(s.Quantity * s.Unit_sale_price) AS Total_Global_Revenue, (SUM(CASE WHEN s.Order_method_code = 5 THEN s.Quantity * s.Unit_sale_price ELSE 0 END) / SUM(s.Quantity * s.Unit_sale_price)) * 100 AS Web_Penetration_Rate FROM `go_explore_prod.daily_sales` AS s;
